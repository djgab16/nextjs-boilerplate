# Kiro Hub MVP: 7-page educational platform (v2)

This second review pass verifies fixes to the 7 issues flagged in v1. Six of the seven concerns were addressed: TutorialCard respects reduced motion, the search dialog uses a proper React context, dead footer links point to `#`, community dates generate relative to today, child page title suffixes are removed, and feature cards no longer inject unnecessary tab stops. Issue 7 (community page filtering) was intentionally deferred as acceptable for MVP scope.

**Watch for:** The `getFutureDate()` approach in the community data resolves the stale-date concern but introduces a build-time coupling — dates are computed once during static generation and will drift back into the past if the site isn't rebuilt periodically (possible, acceptable for MVP).

**Verdict**: APPROVED

## High-level view

All six fixes land cleanly. `TutorialCard` now follows the same `useReducedMotion` guard pattern as every other animated component, with both `initial` and `animate` props conditionally set. The search architecture was reworked from a synthetic `KeyboardEvent` dispatch to a `SearchContext` that shares `{open, setOpen}` between Header and SearchDialog, eliminating the cross-component fragility. Footer dead links to `/accessibility` and `/privacy` now use `href="#"` as inert placeholders. Community event dates use a `getFutureDate(n)` helper that computes relative dates at render time, solving the immediate staleness without requiring dynamic data. Page titles are now bare names (`"Tutorials"`, `"Features"`, `"What is Kiro?"`) that compose correctly with the root layout's `"%s | Kiro Hub"` template. Feature cards dropped both `tabIndex={0}` and the redundant `role="article"`, retaining just the semantic `<motion.article>` with `aria-label`.

The build passes cleanly with all 10 routes (7 pages + sitemap + robots + not-found).

## Community date static-generation coupling

The `getFutureDate` helper uses `new Date()` and `setDate()` to produce ISO date strings offset 14–45 days from "now." Since the community page renders as a static Server Component (confirmed by build output showing `○` for `/community`), "now" is build time. This creates an implicit rebuild frequency requirement — for a marketing MVP behind any CI/CD pipeline that rebuilds on push, the dates stay fresh, but a site deployed once and left unattended for months will show stale events.
