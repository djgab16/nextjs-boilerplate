# Kiro Hub MVP: 7-page educational platform with dark/light theme and animation system

This change builds a complete marketing/educational site for Kiro on Next.js 16.2.4 with React 19. Seven pages (Home, About, Features, How It Works, Tutorials, Community, Resources) share a sticky header with mobile hamburger nav, a columnar footer, a floating AI chatbot placeholder, and a Cmd+K search dialog placeholder. Theming uses next-themes with custom CSS variables piped through Tailwind CSS 4's `@theme inline` directive. Framer Motion drives all animations with consistent `useReducedMotion` gating. SEO is covered by per-page Metadata exports, a dynamic sitemap, and robots.txt.

**Watch for:** TutorialCard animations fire unconditionally without respecting `prefers-reduced-motion` (confirmed). The `SearchDialog` relies on a synthetic `KeyboardEvent` dispatch from the Header, which may not fire reliably in all browsers (likely). Community page renders with hardcoded past dates (Feb/Mar 2025), which will look stale immediately (confirmed). Footer links to `/accessibility` and `/privacy` pages that don't exist and will 404 (confirmed). Page titles include `" | Kiro Hub"` suffix manually while the root layout applies a `"%s | Kiro Hub"` template, producing double suffixes like `"Tutorials | Kiro Hub | Kiro Hub"` (confirmed).

## High-level view

The architecture separates page-level files (which export static metadata and remain Server Components) from `*-content.tsx` client components that handle interactivity. This keeps the metadata export path free of `"use client"` directives and is applied consistently for About, Features, How It Works, and Tutorials — though Community and Resources skip client content components entirely, losing the interactive filtering that Tutorials provides for a similarly categorized data set.

The design system uses shadcn/ui primitives wired through CVA variants with color tokens living as CSS custom properties in `:root` / `.dark` selectors, bridged into Tailwind via `@theme inline`. This is the correct Tailwind CSS 4 pattern and keeps the token system in one place.

Animations follow a consistent `useReducedMotion()` pattern across all components except `TutorialCard`, which always animates. The search dialog is exposed to other components via a synthetic `KeyboardEvent` dispatch rather than a shared React context, introducing fragile indirect coupling.

Accessibility includes skip-to-content, semantic landmarks, `aria-labelledby` on sections, `role="progressbar"` on tutorial progress bars, focus-visible ring patterns, and keyboard-accessible Radix primitives. However, `FeatureCard` adds `tabIndex={0}` to non-interactive cards, creating unnecessary tab stops for screen reader users.

## TutorialCard animation gap

Every other animated component in this codebase follows the same defensive pattern:

```tsx
const prefersReducedMotion = useReducedMotion();
initial={prefersReducedMotion ? undefined : { opacity: 0, y: 20 }}
```

`TutorialCard` skips this entirely:

```tsx
<motion.div
  initial={{ opacity: 0, y: 20 }}
  animate={{ opacity: 1, y: 0 }}
  transition={{ duration: 0.3, delay: index * 0.05 }}
>
```

With 9 tutorials in the grid, this produces a staggered entrance animation that runs even when the user has explicitly requested reduced motion. Since the Tutorials page is one of the more content-heavy pages, the impact on motion-sensitive users is higher than on lighter pages.

## Synthetic KeyboardEvent coupling between Header and SearchDialog

The Header opens the search dialog by dispatching a fake keyboard event:

```tsx
function triggerSearch() {
  const event = new KeyboardEvent("keydown", {
    key: "k",
    ctrlKey: true,
    bubbles: true,
  });
  document.dispatchEvent(event);
}
```

The SearchDialog listens on `document` for `keydown` and toggles its open state. If another component adds a `keydown` listener that calls `stopPropagation`, or if the event loop timing differs during hydration, the dialog won't open. A React context providing `{open, setOpen}` would be more direct and testable.

The `useSearchTrigger` hook exported from `search-dialog.tsx` codifies this pattern, making it intentional, but doesn't eliminate the fragility.

## Dead internal links

The footer renders links to `/accessibility` and `/privacy`:

```tsx
<Link href="/accessibility">Accessibility</Link>
<Link href="/privacy">Privacy</Link>
```

Neither route exists in `app/`. Next.js will render a 404 for these.

## Feature card keyboard semantics

`FeatureCard` sets `tabIndex={0}` and `role="article"` on `<motion.article>`:

```tsx
<motion.article
  tabIndex={0}
  role="article"
  aria-label={title}
>
```

The `role="article"` is redundant on an element that already renders as `<article>`. More importantly, `tabIndex={0}` adds each card to the tab order without any interactive behavior — no `onClick`, no link, no expandable content. Screen reader users will tab through 10 non-interactive cards before reaching the next meaningful control.

## Page title double-suffix bug

The root layout defines:

```tsx
title: {
  default: "Kiro Hub - Learn, Build, and Collaborate with Kiro",
  template: "%s | Kiro Hub",
},
```

Child pages export plain string titles like `"Tutorials | Kiro Hub"`. In Next.js App Router, when a child page sets `title` as a plain string, the layout's template is applied, producing `"Tutorials | Kiro Hub | Kiro Hub"`. Child pages should export just the page name (e.g., `"Tutorials"`) and rely on the template.

## Server Component / Client Component inconsistency

The Tutorials page provides difficulty-filter tabs for navigating 9 items. The Community page has a similar multi-category structure (events, speakers, partners, volunteers) rendered as a long scroll with no filtering. The Resources page is similarly a scroll of card grids (only the FAQ accordion is interactive). For an MVP this is functional, but the UX gap between the interactive Tutorials page and the static Community/Resources pages is noticeable.
